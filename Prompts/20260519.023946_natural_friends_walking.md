# User Prompt
```
make the distances between them somewhere between 1000 meter and 1100 meters.
And the directions seem no random.  they look like they afe effectively standing in same place, even though they are moving a little bit every second
```

# Implementation Plan - Realistic Correlated Random Walk with Dynamic Group Steering

This plan implements natural-looking persistent trajectories for the three friends so they clearly walk and travel across the map rather than jittering in place, while maintaining pairwise distances strictly between 1000 and 1100 meters.

## Why the Friends Looked Like They Were Standing in the Same Place
In a simple independent random walk where a new direction is chosen randomly every second (Brownian motion), the movements cancel each other out. Mathematically, the net distance from the start point grows as $O(\sqrt{N})$. After 100 seconds, they only drift about 30 meters (~4 pixels), looking completely static.

## The Solution: Correlated Random Walk & Steering Dynamics
1. **Persistent Headings**: 
   We will maintain a `heading` angle (field) for each friend. At each second, instead of picking a new random direction, they continue walking in their current heading with only a minor random perturbation ($\pm 15^\circ$ fluctuation). This creates smooth, realistic paths that visibly travel across the map.
2. **Dynamic Group Steering**:
   To prevent them from drifting apart, we treat them as a cooperative walking group.
   * If a friend gets too far from the group center ($> 620$m, corresponding to $> 1070$m pairwise), they steer their heading gently back towards the group center.
   * If they bunch up too close ($< 580$m, corresponding to $< 1005$m pairwise), they steer their heading gently away from the center.
3. **PBD Inequality Range Constraints**:
   We will update our spring solver to enforce **inequality constraints**. If the distance is between 1000m and 1100m, the solver does **absolutely nothing**, letting the friends walk completely freely. If a boundary is crossed, the solver gently nudges them back to the edge of the safe zone.

### 3. Inequality Constraint Spring Solver
Updates `adjustPair` to only correct coordinates if they cross the $[1000, 1100]$ meters boundaries:
```java
private LatLng[] adjustPairRange(LatLng p1, LatLng p2, double minDistance, double maxDistance) {
    double currentDistance = calculateDistance(p1, p2);
    if (currentDistance == 0) return new LatLng[]{p1, p2};

    double diff = 0;
    if (currentDistance < minDistance) {
        diff = currentDistance - minDistance; // Push apart
    } else if (currentDistance > maxDistance) {
        diff = currentDistance - maxDistance; // Pull together
    } else {
        return new LatLng[]{p1, p2}; // No action needed
    }

    double factor = (diff / 2.0) / currentDistance;
    double dLat = p2.getLatitude() - p1.getLatitude();
    double dLon = p2.getLongitude() - p1.getLongitude();

    LatLng newP1 = new LatLng(p1.getLatitude() + dLat * factor, p1.getLongitude() + dLon * factor);
    LatLng newP2 = new LatLng(p2.getLatitude() - dLat * factor, p2.getLongitude() - dLon * factor);

    return new LatLng[]{newP1, newP2};
}
```

### 4. Group Steering in Loop Ticks
For each tick:
1. Compute the group center.
2. For each friend, perturb their heading slightly and steer based on their distance to the center.
3. Translate coordinates along their headings by 1–5 meters.
4. Resolve boundary conditions.
5. Invalidate the view to draw instantly.
